Verum

Websites stored as Ethereum calldata, verified by a light client in your browser.

Why calldata?

Because it is the cheapest place to put bytes on Ethereum that the chain still commits to, and because it is provable without trusting anyone.

Every transaction is a leaf in its block's transaction trie, and the root of that trie is in the block header. So a page's bytes can be proven to belong to a specific transaction, in a specific block, on the canonical chain — by rebuilding the trie locally and checking the root against a header that a light client verified against the beacon chain's sync committee. No indexer, no gateway, no server is trusted at any point.

Contract storage would also give you that, but you pay for a feature you do not need: storage is readable by other contracts. A website is never read by the EVM. Calldata is data the chain agrees on but the EVM never has to keep around — which is exactly what a static site is.

Why not ERC-4804 / web3:// ?

web3:// (ERC-4804, refined in ERC-6860) defines URLs that resolve to a smart-contract call. The bytes usually live in contract bytecode (the SSTORE2 / EthFS pattern: deploy data as a contract, read it back with EXTCODECOPY). It is a good standard and solves a broader problem — Verum makes a narrower trade.

  • Cost. Bytecode costs an extra 200 gas per byte on top of the calldata you already paid to submit it — ~5.4× more expensive than calldata today, and still ~3.4× after Glamsterdam (see below).
  • Verification. Reading a web3:// URL means an eth_call — you need verified state, not just a verified transaction. Most web3:// access today goes through a gateway, and a gateway is a trusted server. Verum verifies transaction inclusion, which needs only block headers.
  • Size limits. A contract caps at 24 KB (EIP-170), so anything larger is split across many contracts. Verum splits across transactions instead, which is the same idea without the deployment overhead.

The honest trade: contract-stored data is readable on-chain. Calldata is not. If your content must be consumed by another contract, use the ERC. If it is a website, it never will be.

Why not IPFS?

IPFS gives you content addressing — a CID proves the bytes are the bytes. What it does not give you is persistence or a canonical answer to "which version is live".

  • Availability is somebody's job. A CID resolves only while someone pins it. Ethereum replicates your bytes across every node that keeps history, as a side effect of consensus — nobody has to volunteer.
  • Most IPFS is really a gateway. In practice content is fetched through ipfs.io or Cloudflare, and then you are trusting an HTTPS server again — the exact thing the CID was supposed to remove.
  • Naming still lands on-chain. The usual setup is an ENS record pointing at a CID — so you already depend on Ethereum for the pointer. Verum drops the second system and points at the data directly.
Verum is not "better than IPFS" — it is a different bet. IPFS optimises for large content and cheap distribution. Verum optimises for small content that must be verifiable and permanent without asking anyone to keep hosting it.

What data?

A transaction to a burn address (0x…57334653 — "W3FS" in ASCII, no known private key), carrying zero ETH, whose calldata is the file. The payload is gzipped and prefixed with a small header:

  • magic W3FS, version, content-type, compression
  • chunk index / total — files larger than one transaction split across several
  • the compressed bytes

A whole site (HTML, CSS, JS, images) is packed into a single bundle, gzipped, then split into ~125 KB chunks — one transaction each. The URL is just the coordinates: w3://<block>:<txIndex>. A name (.eth or .gwei) is optional sugar: a w3 text record holding [[block, txIndex], …].

Nothing is written to contract storage. Nothing is registered anywhere. Deployment is one transaction that stores nothing and calls nothing.

What does it cost to deploy?

Gas per byte of gzipped data. Calldata is getting more expensive: Pectra (EIP-7623) introduced a floor price for transactions that carry data but do no work, taking calldata from 16 to 40 gas per byte. Glamsterdam raises that floor again — EIP-7976 sets it to 64 gas per calldata byte, zero and non-zero alike, to cap worst-case block size and push bulk data toward blobs.

Calldata gets 60% dearer; bytecode is untouched, because at 200 gas per byte of code deposit its standard cost already exceeds any floor. So the gap narrows from 5.4× to 3.4×. Calldata stays the cheaper option — Glamsterdam just costs it some of its lead, exactly as Pectra did.

The direction of travel is not in doubt: calldata keeps getting pricier, and every such repricing narrows — without closing — the gap to contract storage.

What is the catch?

Two, stated plainly.

  • History expiry (EIP-4444). Nodes are expected to stop serving old history. Calldata is not lost — it is in era files, archives and the Portal Network — but fetching a years-old page may mean going to those, not to an RPC. Verum already does this: era files, the Portal Network, and a beacon-chain Merkle proof back to a light-client-verified anchor.
  • Contracts cannot read it. Calldata is invisible to the EVM. This is precisely why it is cheap, and precisely why the ERC exists. For a website it costs you nothing.
Verum is a Chrome extension. It verifies every byte it renders against Ethereum consensus using a Helios light client — no RPC, gateway, or indexer is trusted. This page is itself stored as calldata on Ethereum mainnet.
