Verum

Data and websites stored as Ethereum calldata, verified by a light client in your browser.

Why calldata?

Calldata is the cheapest data source on the Ethereum blockchain. It also allows submitting large amounts of data per txt (up to txt gas limit). It is periodically pruned by non archive node, is only processed during transaction execution and is not persisted in the network's "state". Submitting large amounts of data to the network therefore incurs a significantly lower permanent storage burden.

Calldata can be retrieved from local or external archive execution nodes and cryptographically verified against the Ethereum blockchain. The dependencie on archive nodes (or the portal network) indirectly incentivises users to run their own archive node for availability guarantees. Potentially strengthening the Ethereum network.

Why not ERC-4804?

web3:// (ERC-4804, refined in ERC-6860) defines URLs that resolve to a smart-contract call. Since the data is stored as calldata ERC-4804 does not support it.

Why not IPFS?

IPFS gives you content addressing — a CID proves the bytes are the bytes. What it does not give you is persistence or a canonical answer to "which version is live".

  • Availability is somebody's job. A CID resolves only while someone pins it. Ethereum replicates your bytes across every node that keeps history — nobody has to volunteer.
  • Naming still lands on-chain. When using an ENS record pointing at a CID you already depend on Ethereum for the pointer. Verum drops the second system and points at the data directly.

What data?

Any data can be deployed. The payload is gzipped and prefixed with a small header:

  • magic W3FS, version, content-type, compression
  • chunk index / total — files larger than one transaction split across several
  • the compressed bytes

Verum loads websites (including Dapps) in the browsers. Known file types are also loaded in the browser. Unknown file types are provided with a download link.

Effect of history expiry (EIP-4444)?

Nodes are expected to stop serving old history as part of History expiry (EIP-4444). Calldata is not lost. It is in era files, archives and the Portal Network. Fetching a years-old page may mean going to those, not to an RPC. Verum already supports: era files, the Portal Network, and a local archive node.

Verum is a Chrome extension. It verifies every byte it renders against Ethereum consensus using a Helios light client — no RPC, gateway, or indexer is trusted. When running a portal network node or local archive node in addition to verified the data is guaranteed available to the user. This page is itself stored as calldata on Ethereum mainnet.
